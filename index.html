Le concept

Une alarme. Deux façons de la déclencher.

Un son strident, plein écran, qui force le volume et passe outre le mode silencieux. Impossible à manquer.

🎲

Mode entraînement

L'alarme se déclenche 1 à 4 fois par an, à un instant imprévisible, dans une plage horaire que vous choisissez. Personne ne sait quand : on mesure la vraie réactivité, pas l'exercice répété par cœur.

Chrono de réaction intégré
⚡

Mode incident réel

Votre supervision détecte un problème grave et appelle l'API de déclenchement : l'alarme retentit instantanément sur les téléphones de l'équipe d'astreinte. Le drame ne dort jamais — votre alerte non plus.

Webhook · temps réel
Cas d'usage

Du bâtiment à la salle serveurs.

La même mécanique d'alerte, qu'il s'agisse de protéger des personnes ou des données.

🔥

Évacuation incendie

L'exercice surprise classique, mais réellement imprévisible. On chronomètre le temps d'évacuation réel de chacun.

🖥️

Serveur en panne

Le serveur de production tombe à 3 h du matin. Qui réagit, en combien de temps ? Entraînez l'astreinte, ou alertez à l'instant T.

🔒

Ransomware / disque chiffré

Une détection de chiffrement anormal déclenche l'alerte : chaque minute compte pour isoler les machines avant la propagation.

🗄️

Base de données supprimée

Un DROP de trop, une réplication corrompue. L'alarme lance la procédure de restauration d'urgence, sans délai.

Fonctionnement

Quatre étapes, zéro friction.

Installez l'app

Sur les téléphones de l'équipe (Android). Quelques réglages, et c'est prêt.

Choisissez le mode

Entraînement planifié (fréquence + plage horaire) ou déclenchement via l'API.

Ça sonne

Son strident, plein écran rouge, vibration — même app fermée ou téléphone verrouillé.

On mesure

Un chrono démarre. « Mettre en sourdine » puis « Exercice terminé » : le temps de réaction est figé.

API de déclenchement

Faites sonner l'alerte par webhook.

Un appel HTTP depuis votre supervision (Zabbix, Prometheus, un cron, un script de détection…) suffit à réveiller l'astreinte.

⚠️ Spécification en cours de conception. Cette page définit le contrat d'API visé ; le backend et la réception côté application seront implémentés ensuite.
POST

https://<votre-domaine>/api/v1/trigger
ChampTypeDescription
typestringfire, server_down, ransomware, db_loss, custom
severitystringinfo · warning · critical
titlestringTitre affiché en plein écran.
messagestringDétail de l'incident.
targetsstring[]Identifiants d'appareils / groupes d'astreinte (optionnel = tous).
sourcestringOrigine de l'alerte (ex. zabbix-prod).
Requête — exemple curl
curl -X POST https://votre-domaine/api/v1/trigger \
  -H "Authorization: Bearer <API_KEY>" \
  -H "Content-Type: application/json" \
  -d '{
    "type": "ransomware",
    "severity": "critical",
    "title": "Chiffrement anormal détecté",
    "message": "Pic d'écritures sur NAS-01 — isoler immédiatement.",
    "targets": ["astreinte-infra"],
    "source": "edr-sensor-12"
  }'
Réponse — 202 Accepted
{
  "id": "trg_9f2a7c",
  "status": "dispatched",
  "delivered_to": 3,
  "received_at": "2026-06-24T23:14:05Z"
}

// Authentification par clé API (Bearer). Les appareils s'enregistrent depuis l'app.
// Acquittement : POST /api/v1/triggers/{id}/ack — pour mesurer le temps de réaction.
